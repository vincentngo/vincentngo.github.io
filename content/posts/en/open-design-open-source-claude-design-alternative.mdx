## What Is Open Design?

Open Design (OD) is an **open-source alternative to Anthropic's Claude Design**. Released under Apache-2.0, it provides the same artifact-first design loop â€” natural language brief â†’ interactive question form â†’ live todo stream â†’ sandboxed HTML preview â†’ export â€” but without the closed-source walls.

The project describes itself as *"local-first, web-deployable, BYOK at every layer."* It auto-detects **11 coding-agent CLIs** already on your `PATH`, drives them through **34+ composable Skills**, and ships with **129 brand-grade Design Systems**. No CLI installed? An OpenAI-compatible BYOK proxy lets you paste any `baseUrl` + `apiKey` + `model` and still run the full loop.

## Unique Features Worth Calling Out

### 1. The Interactive Question Form Prevents 80% of Redirects

OD's prompt stack hard-codes **Rule 1**: every fresh design brief begins with a `<question-form id="discovery">` instead of code. Surface Â· audience Â· tone Â· brand context Â· scale Â· constraints. Thirty seconds of radio buttons beats thirty minutes of "can you make it more modern?" redirects.

### 2. The Five-Dimensional Self-Critique Gate

Before emitting any `<artifact>`, the agent silently scores its output 1â€“5 across:
- **Philosophy** â€” does it match the chosen direction?
- **Visual Hierarchy** â€” does the eye know where to go first?
- **Detail & Execution** â€” are the margins consistent? Are the colors from the system?
- **Functionality** â€” does it work at the requested fidelity?
- **Innovation & Restraint** â€” is it interesting without being chaotic?

Anything under 3/5 is a regression â€” fix and rescore. Two passes is normal.

### 3. The Anti-AI-Slop Machinery

Inspired by `huashu-design`, the prompt stack explicitly forbids:
- Aggressive purple gradients
- Generic emoji icons
- Rounded cards with left-border accents
- Hand-drawn SVG humans
- Inter as a *display* face
- Invented metrics like "10Ă— faster"

Honest placeholders (`â€”` or grey blocks) are preferred over fake stats.

### 4. Claude Design ZIP Import

Drop an export from `claude.ai` onto the welcome dialog. OD parses it into a real project folder and stages a *"continue where Anthropic left off"* prompt for your local agent. No re-prompting required.

### 5. Real Filesystem Workspace

The daemon spawns the CLI with `cwd` set to the project's artifact folder under `.od/projects/<id>/`. The agent gets real `Read`, `Write`, `Bash`, `WebFetch` against a real on-disk environment. It can `grep` your CSS for hex values, write a `brand-spec.md`, drop generated images, and produce `.pptx` / `.zip` / `.pdf` files that show up as download chips.

### 6. Artifact Lint API

`POST /api/artifacts/lint` runs structural checks on generated artifacts (broken `<artifact>` framing, missing required side files, stale palette tokens) and returns findings the agent can read back into its next turn.

### 7. Tab Persistence & SQLite State

Projects, conversations, messages, and open tabs persist in a local SQLite DB at `.od/app.sqlite`. Reopen tomorrow â€” the todo card and open files are exactly where you left them.

## Why Startups Should Care (The Cost & Lock-In Argument)

I run a startup. Every dollar and every dependency is a decision that compounds. Here's why Open Design matters beyond just "it's cheaper":

### 1. No Separate Usage Quota

Claude Design has its own usage limits *on top of* your regular Anthropic subscription. Reviewers report burning 15% of a weekly budget on a single landing page. Open Design uses your regular model tokens. If you're already paying for Claude Code, Codex, or Cursor â€” you're already paying for the design engine.

### 2. Model Portability

Today Anthropic has the best design model. Tomorrow it might be OpenAI, Google, or a local Qwen 3.6 35B-A3B running on your M2 Max via oMLX. Open Design doesn't care. Swap the CLI, keep the skills, keep the design systems, keep your projects.

### 3. Skills as Files, Not Plugins

A skill is a folder with a `SKILL.md` + `assets/` + `references/`. Version it in git. Fork it. Share it with your team as a repo. Run your company's branded deck skill without rebuilding a product. This is infrastructure, not a feature.

### 4. Design Systems as Pull Requests

A `DESIGN.md` is a text file you can review in GitHub. Your brand tokens, component rules, and anti-patterns live in version control, not in an ephemeral chat thread. Onboard a new designer â€” point them at the `DESIGN.md`. Onboard a new agent â€” it reads the same file.

### 5. Local-First = Data Sovereignty

Your briefs, your artifacts, your brand specs, your conversation history â€” all on your machine in `.od/`. No cloud egress. No training-data ambiguity. No "we updated our terms of service."
